# 入口与配置

## llm.py — 用户入口

```python
from nanovllm.engine.llm_engine import LLMEngine

class LLM(LLMEngine):
    pass
```

这个仅 **82 字节** 的文件是整个项目的用户入口。`LLM` 直接继承 `LLMEngine` 而不添加任何代码，纯语义命名——用户用 `LLM.generate()` 比 `LLMEngine.generate()` 更自然。这种"入口即别名"的设计在保持架构清晰的同时提供直观的 API。

## sampling_params.py — 采样参数

```python
@dataclass(slots=True)
class SamplingParams:
    temperature: float = 1.0
    max_tokens: int = 64
    ignore_eos: bool = False

    def __post_init__(self):
        assert self.temperature > 1e-10, "greedy sampling is not permitted"
```

简单到极致的采样参数设计：

- `temperature`：控制随机性，用 `> 1e-10` 确保不会出现除零错误（采样器用 `logits / temperature`）。注意项目**不支持 greedy decoding**（temperature=0），这是一项已知限制
- `max_tokens`：每个请求最多生成的 token 数
- `ignore_eos`：benchmark 模式下忽略 EOS token，始终生成 max_tokens 个 token
- `slots=True`：减少内存占用，因为每个请求都会创建一个 SamplingParams 副本

## config.py — 全局配置

```python
@dataclass(slots=True)
class Config:
    model: str                          # 模型路径（必须）
    max_num_batched_tokens: int = 16384 # 每 batch 最大 token 数
    max_num_seqs: int = 512             # 每 batch 最大序列数
    max_model_len: int = 4096          # 最大上下文长度（会被 hf_config 限制）
    gpu_memory_utilization: float = 0.9 # GPU 显存利用率
    tensor_parallel_size: int = 1      # 张量并行 GPU 数量（1-8）
    enforce_eager: bool = False        # True 时禁用 CUDA Graph
    hf_config: AutoConfig | None = None # HuggingFace 模型配置（自动加载）
    eos: int = -1                       # EOS token id（从 tokenizer 自动设置）
    kvcache_block_size: int = 256      # KV-Cache 块大小（必须能被 256 整除）
    num_kvcache_blocks: int = -1       # KV-Cache 块数（运行时自动计算）
```

### `__post_init__` 的初始化逻辑

```python
def __post_init__(self):
    assert os.path.isdir(self.model)                        # 模型路径必须存在
    assert self.kvcache_block_size % 256 == 0               # 块大小对齐保证
    assert 1 <= self.tensor_parallel_size <= 8              # TP 范围限制
    self.hf_config = AutoConfig.from_pretrained(self.model) # 自动推断模型配置
    self.max_model_len = min(self.max_model_len,
                             self.hf_config.max_position_embeddings)
```

关键点：
- `kvcache_block_size % 256 == 0` 的约束来自注意力 kernel 的对齐要求
- `max_model_len` 会被 hf_config 的实际值截断，避免无效请求
- `num_kvcache_blocks` 初始为 -1，在 `ModelRunner.allocate_kv_cache()` 中根据 GPU 显存动态计算

### 从用户 kwargs 到 Config 的转换

在 `LLMEngine.__init__()` 中：
```python
config_fields = {field.name for field in fields(Config)}
config_kwargs = {k: v for k, v in kwargs.items() if k in config_fields}
config = Config(model, **config_kwargs)
```

这个设计让用户可以用自然的方式传递参数（如 `LLM(path, enforce_eager=True)`），而 `Config` dataclass 提供类型安全和默认值，`fields(Config)` 过滤确保不会有无效参数进入。这是一种优雅的"用户 kwargs → 内部配置"桥接模式。
