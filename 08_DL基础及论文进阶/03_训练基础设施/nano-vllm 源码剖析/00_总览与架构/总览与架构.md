# 总览与架构

## nano-vllm 的总体架构

nano-vllm 的系统架构可以分为 **四层**：

```
┌──────────────────────────────────────────────────────┐
│                    LLM (用户入口)                       │
│  - llm.generate(prompts, sampling_params)             │
│  - 继承自 LLMEngine，无额外代码                          │
└──────────────────────────────────────────────────────┘
                            │
                            ▼
┌──────────────────────────────────────────────────────┐
│              LLMEngine (引擎主循环)                      │
│  - generate()：等待+收集结果 + tqdm 进度条               │
│  - add_request()：接收请求，创建 Sequence                  │
│  - step()：单步调度 → 推理 → 后处理                       │
│  - exit()：清理进程和资源                               │
└──────────────────────────────────────────────────────┘
          │                        │
          ▼                        ▼
┌──────────────────┐    ┌──────────────────────────────┐
│   Scheduler       │    │   ModelRunner (GPU 推理引擎)     │
│  - 连续批处理       │    │  - Prefill/Decode 状态准备       │
│  - 抢占策略         │    │  - CUDA Graph 捕获与回放       │
│  - 前缀缓存协调      │    │  - Tensor Parallel 管理      │
│  - 序列生命周期管理   │    │  - warmup + KV-Cache 分配    │
└──────────────────┘    └──────────────────────────────┘
          │                        │
          ▼                        ▼
┌──────────────────┐    ┌──────────────────────────────┐
│  BlockManager     │    │  Qwen3ForCausalLM (模型)       │
│  - KV-Cache 块管理 │    │  - Qwen3Model → DecoderLayer │
│  - 前缀哈希匹配     │    │  - Attention + MLP + RMSNorm │
│  - Ref-Count 复用 │    │  - ParallelLMHead            │
└──────────────────┘    └──────────────────────────────┘
```

## 请求生命周期

一个请求从进入到输出 token 的完整路径：

```
1. 用户调用 llm.generate(prompts, sampling_params)
2. LLMEngine.add_request() 创建 Sequence，加入 Scheduler.waiting 队列
3. Scheduler.schedule() 决定本轮执行哪些序列
   ├─ Prefill: 从 waiting 队列取，分配/复用 KV-Cache Block
   └─ Decode: 从 running 队列取，逐个追加 token
4. ModelRunner.run() 执行 GPU 推理
   ├─ prepare_prefill() / prepare_decode() 准备输入
   ├─ run_model() 执行前向推理（可能走 CUDA Graph）
   └─ sampler() 采样输出 token
5. Scheduler.postprocess() 更新序列状态
   ├─ 追加 token，检查 EOS / max_tokens
   └─ 完成序列释放 KV-Cache Block，加入输出列表
6. LLMEngine.generate() 收集结果，tokenizer.decode() 返回文本
```

## 关键技术栈

| 技术 | 实现位置 | 说明 |
|------|----------|------|
| PagedAttention | block_manager.py + attention.py | KV-Cache 按 256-token block 管理，用 block_table 映射 |
| Continuous Batching | scheduler.py | 同一 batch 混合 Prefill 和 Decode |
| Prefix Caching | block_manager.py | xxhash 计算 block 哈希，匹配前缀共享 KV-Cache |
| Tensor Parallelism | model_runner.py + linear.py + embed_head.py | SharedMemory + dist.gather/all_reduce |
| CUDA Graph | model_runner.py capture_cudagraph() | Decode 阶段的 batch_size 分档捕获 |
| FlashAttention | attention.py | flash_attn_varlen_func / flash_attn_with_kvcache |
| Torch Compile | layernorm.py + activation.py + sampler.py | 小算子编译加速 |
| Triton Kernel | attention.py store_kvcache_kernel | 自定义 KV-Cache 存储 kernel |

## 代码量统计

| 模块 | 行数 | 占比 |
|------|------|------|
| engine/model_runner.py | ~340 | 28% |
| engine/block_manager.py | ~120 | 10% |
| layers/linear.py | ~145 | 12% |
| models/qwen3.py | ~220 | 18% |
| engine/llm_engine.py | ~110 | 9% |
| engine/scheduler.py | ~105 | 9% |
| 其余文件 | ~160 | 13% |
| **总计** | **~1200** | 100% |

最核心的代码集中在 `model_runner.py`（GPU 推理生命周期）和 `linear.py`（张量并行线性层），其次是模型定义和块管理器。
