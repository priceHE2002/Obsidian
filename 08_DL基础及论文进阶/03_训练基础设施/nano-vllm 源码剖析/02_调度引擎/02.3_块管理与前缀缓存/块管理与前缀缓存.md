# 块管理与前缀缓存

`BlockManager` 是 nano-vllm 最精妙的部分之一，它实现了 PagedAttention 的两大核心能力：KV-Cache 块管理和基于哈希的前缀缓存。

## Block — KV-Cache 的基本单元

```python
class Block:
    def __init__(self, block_id):
        self.block_id = block_id      # 全局唯一块 ID
        self.ref_count = 0            # 引用计数：多个序列共享
        self.hash = -1                # 内容哈希（用于前缀匹配）
        self.token_ids = []           # 块对应的 token_ids（用于哈希验证）
```

每个 Block 存储 `block_size`（默认 256）个 token 的 KV-Cache。`ref_count` 是前缀缓存的基石：多个序列共享同一个前缀时，对应的 Block 的 `ref_count` > 1，只有所有共享序列都释放后才真正回收。

## BlockManager 数据结构

```python
class BlockManager:
    def __init__(self, num_blocks, block_size):
        self.block_size = block_size
        self.blocks: list[Block] = [Block(i) for i in range(num_blocks)]
        self.hash_to_block_id: dict[int, int] = dict()        # 哈希 → Block ID 映射
        self.free_block_ids: deque[int] = deque(range(num_blocks))
        self.used_block_ids: set[int] = set()
```

三层索引结构：
- `blocks`：按 ID 直接寻址所有 Block 对象
- `hash_to_block_id`：通过 xxhash 快速查找前缀匹配
- `free_block_ids` / `used_block_ids`：O(1) 分配和释放

## 哈希计算

```python
@classmethod
def compute_hash(cls, token_ids, prefix=-1):
    h = xxhash.xxh64()
    if prefix != -1:
        h.update(prefix.to_bytes(8, "little"))     # 链式哈希：包含前一个 Block 的哈希
    h.update(np.array(token_ids).tobytes())
    return h.intdigest()
```

这个哈希设计的精妙之处在于**链式哈希（rolling hash）**：计算第 i 个 Block 的哈希时，把第 i-1 个 Block 的哈希也作为输入。这确保了不同上下文中相同 token 序列的哈希是不同的，避免了前缀碰撞。例如：

- 序列 A: "The cat sat" → Block 0 hash = H("The cat sat"), Block 1 hash = H(H0, " on the mat")
- 序列 B: "A big cat" → Block 0 hash = H("A big cat"), Block 1 hash = H(H0', " sat on the") 

虽然 Block 1 中都有 " sat"，但因为前缀哈希不同，两个 Block 1 的哈希也不同——正确地区分了上下文。

## 前缀缓存分配：can_allocate() + allocate()

### can_allocate() — 检查可行性与缓存命中

```python
def can_allocate(self, seq):
    h = -1
    num_cached_blocks = 0          # 命中缓存的 Block 数
    num_new_blocks = seq.num_blocks  # 总共需要的 Block 数

    for i in range(seq.num_blocks - 1):
        token_ids = seq.block(i)
        h = self.compute_hash(token_ids, h)
        block_id = self.hash_to_block_id.get(h, -1)
        if block_id == -1 or self.blocks[block_id].token_ids != token_ids:
            break                     # 缓存未命中，停止搜索
        num_cached_blocks += 1
        if block_id in self.used_block_ids:
            num_new_blocks -= 1       # 已在使用中，不需要新分配

    if len(self.free_block_ids) < num_new_blocks:
        return -1                     # 显存不够
    return num_cached_blocks
```

这段代码逐步计算序列每个 Block 的链式哈希，然后查 `hash_to_block_id` 找匹配的缓存 Block。它的关键在于双重验证——即使哈希命中，还要比较 `token_ids`，防止哈希碰撞。只扫描到 `num_blocks - 1`（最后一个 Block）是因为最后一个 Block 还不完整，不会有缓存命中。

`num_new_blocks` 的计算也很巧妙：已经 `在 used_block_ids` 中的 Block 可以通过引用计数共享，不需要新分配，所以 `num_new_blocks -= 1`。

### allocate() — 执行分配

```python
def allocate(self, seq, num_cached_blocks):
    assert not seq.block_table
    h = -1

    # 阶段 1: 复用缓存 Block
    for i in range(num_cached_blocks):
        token_ids = seq.block(i)
        h = self.compute_hash(token_ids, h)
        block_id = self.hash_to_block_id[h]
        block = self.blocks[block_id]
        if block_id in self.used_block_ids:
            block.ref_count += 1       # 增加引用计数
        else:
            block.ref_count = 1        # 首次使用
            self.free_block_ids.remove(block_id)
            self.used_block_ids.add(block_id)
        seq.block_table.append(block_id)

    # 阶段 2: 分配新 Block
    for i in range(num_cached_blocks, seq.num_blocks):
        seq.block_table.append(self._allocate_block())

    seq.num_cached_tokens = num_cached_blocks * self.block_size
```

`num_cached_tokens` 被设置为缓存命中的 token 数，模型 Prefill 时会跳过这些 token，直接从缓存后的位置开始计算。

## 前缀缓存的完整流程图示

假设序列 A "The cat sat on the mat"（8 tokens，block_size=4）已经完成并获得两个 Block {h0, h1}。现在序列 B "The cat sat on the floor"（8 tokens）到达：

```
序列 B 的 Block:
  Block 0: "The cat sat on" → 哈希 h0匹配！→ 复用 A 的 Block 0 (ref_count=2)
  Block 1: "the floor"       → 哈希 h1'不匹謒 → 分配新 Block 2

序列 B 的 block_table = [A的Block0, Block2]
序列 B 的 num_cached_tokens = 1 * 4 = 4

Prefill 时只需计算 Block 1 的 4 个 token，Block 0 的 KV-Cache 直接复用！
```

## Decode 阶段的块管理

```python
def can_append(self, seq):
    return len(self.free_block_ids) >= (len(seq) % self.block_size == 1)

def may_append(self, seq):
    if len(seq) % self.block_size == 1:
        seq.block_table.append(self._allocate_block())
```

这两个方法处理 Decode 阶段的 Block 分配。`len(seq) % self.block_size == 1` 意味着当前 token 刚好需要一个新的 Block 的第一个 slot，此时需要分配新 Block。这个检查在每轮 Decode 时执行，确保 KV-Cache 始终有空间写入。

## hash_blocks() — 写入缓存

```python
def hash_blocks(self, seq):
    start = seq.num_cached_tokens // self.block_size
    end = (seq.num_cached_tokens + seq.num_scheduled_tokens) // self.block_size
    if start == end: return        # 没有完整的 Block 需要哈希
    h = self.blocks[seq.block_table[start-1]].hash if start > 0 else -1
    for i in range(start, end):
        block = self.blocks[seq.block_table[i]]
        token_ids = seq.block(i)
        h = self.compute_hash(token_ids, h)
        block.update(h, token_ids)
        self.hash_to_block_id[h] = block.block_id
```

这个方法在每轮 `postprocess()` 中调用。它只哈希**刚写满的完整 Block**（从 `num_cached_tokens` 到 `num_cached_tokens + num_scheduled_tokens`），避免了重复工作。链式哈希从上一个 Block 的哈希开始，保持了一致性。

## 释放（deallocate）

```python
def deallocate(self, seq):
    for block_id in reversed(seq.block_table):
        block = self.blocks[block_id]
        block.ref_count -= 1
        if block.ref_count == 0:
            self._deallocate_block(block_id)
    seq.num_cached_tokens = 0
    seq.block_table.clear()
```

倒序释放（`reversed`）是一个细节优化：先释放末端的 Block 可以更快地将连续空间归还给 free list。`ref_count == 0` 时才真正回收——这是前缀共享的生命线。
