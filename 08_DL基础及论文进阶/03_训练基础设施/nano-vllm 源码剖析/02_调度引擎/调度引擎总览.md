# 调度引擎总览

调度引擎是 nano-vllm 的"大脑"，负责决定每一轮推理（step）应该处理哪些序列、分配多少 token、如何复用 KV-Cache。它由三个文件组成：

| 文件 | 职责 | 行数 |
|------|------|------|
| `scheduler.py` | 核心调度逻辑：Prefill/Decode 调度、抢占 | ~105 |
| `sequence.py` | 序列状态机：生命周期、token 管理 | ~75 |
| `block_manager.py` | KV-Cache 块管理：PagedAttention + 前缀哈希 | ~120 |

## 调度器与 BlockManager 的协作

```
Scheduler.schedule()
    │
    ├─ Prefill 阶段：
    │   ├─ block_manager.can_allocate(seq)    ← 检查是否有足够内存 + 前缀缓存命中
    │   ├─ block_manager.allocate(seq, ...)   ← 分配/复用 KV-Cache Block
    │   └─ seq.num_scheduled_tokens = N       ← 设置本轮需要推理的 token 数
    │
    ├─ Decode 阶段：
    │   ├─ block_manager.can_append(seq)      ← 检查是否需要新 Block
    │   └─ block_manager.may_append(seq)      ← 如需新 Block 则分配
    │
    └─ postprocess()：
        ├─ block_manager.hash_blocks(seq)     ← 计算完成的 Block 的哈希
        ├─ seq.append_token(token_id)         ← 追加采样 token
        └─ block_manager.deallocate(seq)      ← 完成的序列释放 Block
```

## 调度策略

nano-vllm 采用 **Preempt-on-Recourse (抢占策略)**：

1. **Prefill 优先**：每轮先调度 Prefill 请求，直到达到 `max_num_batched_tokens` 或 `max_num_seqs`
2. **Chunked Prefill**：当剩余 token 预算不够完整 Prefill 时，只对第一个序列做分块 Prefill（`remaining < num_tokens and scheduled_seqs` 时 break），后续序列等待下一轮
3. **Decode Round-Robin**：Running 队列按 FIFO 取出，Decode 追加后放回队列尾部
4. **抢占**：Decode 时如果内存不够，从 Running 队列尾部抢占（deallocate + 退回 Waiting 队列头部）
