# 调度器核心

`Scheduler` 是整个推理引擎的调度中枢。它管理两类序列队列——`waiting`（等待 Prefill）和 `running`（正在 Decode）——并在每轮 `schedule()` 调用中决定谁获得 GPU 资源。

## 数据结构

```python
class Scheduler:
    def __init__(self, config: Config):
        self.max_num_seqs = config.max_num_seqs            # 最大并行序列数
        self.max_num_batched_tokens = config.max_num_batched_tokens  # 每 batch 最大 token 数
        self.eos = config.eos                               # EOS token id
        self.block_size = config.kvcache_block_size         # KV-Cache 块大小
        self.block_manager = BlockManager(...)              # KV-Cache 块管理器
        self.waiting: deque[Sequence] = deque()             # 等待队列（FIFO）
        self.running: deque[Sequence] = deque()             # 运行队列（循环）
```

选择 `deque` 而不是 `list` 是因为：
- `popleft()` / `appendleft()` 都是 O(1)，适合 FIFO
- `extendleft()` 可以将 decode 后的序列高效放回队列头部保持顺序

## schedule() 方法 — 核心调度逻辑

```python
def schedule(self) -> tuple[list[Sequence], bool]:
    scheduled_seqs = []
    num_batched_tokens = 0

    # 阶段 1: Prefill
    while self.waiting and len(scheduled_seqs) < self.max_num_seqs:
        seq = self.waiting[0]                               # 只取不弹
        remaining = self.max_num_batched_tokens - num_batched_tokens
        if remaining == 0: break

        if not seq.block_table:                              # 首次调度，需要分配 KV-Cache
            num_cached_blocks = self.block_manager.can_allocate(seq)
            if num_cached_blocks == -1: break                # 显存不够
            num_tokens = seq.num_tokens - num_cached_blocks * self.block_size
        else:
            num_tokens = seq.num_tokens - seq.num_cached_tokens  # Chunked Prefill 剩余

        # ⚡ 关键限制：只有第一个序列可以做 Chunked Prefill
        if remaining < num_tokens and scheduled_seqs: break

        if not seq.block_table:
            self.block_manager.allocate(seq, num_cached_blocks)
        seq.num_scheduled_tokens = min(num_tokens, remaining)
        num_batched_tokens += seq.num_scheduled_tokens

        if seq.num_cached_tokens + seq.num_scheduled_tokens == seq.num_tokens:
            seq.status = SequenceStatus.RUNNING
            self.waiting.popleft()   # Prefill 完成，从 waiting 移动到 running
            self.running.append(seq)
        scheduled_seqs.append(seq)

    if scheduled_seqs: return scheduled_seqs, True   # 有 Prefill 序列，本轮是 Prefill

    # 阶段 2: Decode
    while self.running and len(scheduled_seqs) < self.max_num_seqs:
        seq = self.running.popleft()
        while not self.block_manager.can_append(seq):
            if self.running:
                self.preempt(self.running.pop())     # 从 running 尾部抢占
            else:
                self.preempt(seq)                     # 全部抢占，包括自身
                break
        else:
            seq.num_scheduled_tokens = 1
            seq.is_prefill = False
            self.block_manager.may_append(seq)
            scheduled_seqs.append(seq)

    assert scheduled_seqs
    self.running.extendleft(reversed(scheduled_seqs))  # 保持相对顺序
    return scheduled_seqs, False                        # 本轮是 Decode
```

## 关键设计决策分析

### 1. Chunked Prefill：为什么只允许第一个序列？

```python
if remaining < num_tokens and scheduled_seqs: break
```

这个设计选择基于以下权衡：如果当前 batch 中已有序列完成了部分 Prefill，再让新序列做 Chunked Prefill 会导致内部状态复杂化（每个序列的 `num_cached_tokens` 不同，block table 部分分配）。限制为只对第一个序列做 Chunked Prefill 简化了状态管理，且实际影响有限——长 prompt 通常不会多个同时到达。

### 2. 抢占策略解析

```python
def preempt(self, seq: Sequence):
    seq.status = SequenceStatus.WAITING
    seq.is_prefill = True
    self.block_manager.deallocate(seq)  # 释放 KV-Cache
    self.waiting.appendleft(seq)        # 放回 waiting 头部，优先服务
```

抢占时从 running 队列**尾部**开始（最后进入的序列先被抢占），放入 waiting 队列**头部**（被抢占的序列优先重调度）。这是一个公平性设计：被抢占的序列下次优先服务。抢占时 `deallocate` 会释放所有 KV-Cache Block，意味着该序列下次调度时需要完整重新 Prefill。

### 3. Decode 的循环调度

```python
seq = self.running.popleft()         # 取头部
# ... decode ...
self.running.extendleft(reversed(scheduled_seqs))  # 放回头部（保持顺序）
```

这个精巧操作确保 running 队列中的序列以 Round-Robin 方式公平获得 decode 机会：`popleft()` 取出队首，`extendleft(reversed(...))` 保持处理顺序将结果放回队首。

## postprocess() 方法 — 结果处理

```python
def postprocess(self, seqs, token_ids, is_prefill):
    for seq, token_id in zip(seqs, token_ids):
        self.block_manager.hash_blocks(seq)           # 计算完成 Block 的哈希
        seq.num_cached_tokens += seq.num_scheduled_tokens
        seq.num_scheduled_tokens = 0
        if is_prefill and seq.num_cached_tokens < seq.num_tokens:
            continue                                  # Chunked Prefill 未完成，不生成新 token
        seq.append_token(token_id)
        if (not seq.ignore_eos and token_id == self.eos) or \
           seq.num_completion_tokens == seq.max_tokens:
            seq.status = SequenceStatus.FINISHED
            self.block_manager.deallocate(seq)
            self.running.remove(seq)
```

关键点：
- Chunked Prefill 的序列在未完成全部 Prefill 前不会 `append_token`（因为模型还没输出这个位置的 token）
- 完成序列立即释放 KV-Cache，让出显存给等待队列的新请求
- `hash_blocks` 在每个 Block 写满后立即计算哈希，确保前缀缓存实时可用
