# 序列管理

`Sequence` 类负责管理单个推理请求的完整生命周期——从 prompt token 到最终生成 token 的全过程状态。

## 序列状态机

```python
class SequenceStatus(Enum):
    WAITING = auto()    # 等待首次 Prefill 或抢占后等待重新调度
    RUNNING = auto()    # Prefill 完成，正在进行 Decode（或 Chunked Prefill 中）
    FINISHED = auto()   # 生成完成：遇到 EOS 或达到 max_tokens
```

状态转换路径：

```
WAITING ──(首次 Prefill 完成)──▶ RUNNING
   ▲                              │
   │  (被抢占)                      │
   └──────────────────────────────┘
                                  │
                            (EOS / max_tokens)
                                  │
                                  ▼
                              FINISHED
```

## Sequence 核心属性

```python
class Sequence:
    block_size = 256                    # 类变量，从 config 设置
    counter = count()                   # 类变量，全局递增 ID 生成器

    def __init__(self, token_ids, sampling_params=SamplingParams()):
        self.seq_id = next(Sequence.counter)           # 全局唯一 ID
        self.status = SequenceStatus.WAITING
        self.token_ids = copy(token_ids)               # 深拷贝，避免外部修改
        self.last_token = token_ids[-1]
        self.num_tokens = len(self.token_ids)          # 总 token 数（prompt + 已生成的）
        self.num_prompt_tokens = len(token_ids)        # prompt 部分的 token 数
        self.num_cached_tokens = 0                     # 已缓存（已 Prefill）的 token 数
        self.num_scheduled_tokens = 0                  # 本轮调度要处理的 token 数
        self.is_prefill = True                         # 当前是否处于 Prefill 状态
        self.block_table = []                          # KV-Cache Block ID 列表
        self.temperature = sampling_params.temperature
        self.max_tokens = sampling_params.max_tokens
        self.ignore_eos = sampling_params.ignore_eos
```

### 属性关系图

```
token_ids = [prompt tokens ... | generated tokens ...]
            ├─ num_prompt_tokens ─┤├─ num_completion_tokens ─┤
            └─         num_tokens (total)                ────┘

num_cached_tokens: 已 Prefill 完成的 token 数（≤ num_tokens）
num_scheduled_tokens: 本轮计划处理的 token 数
```

### 关键计算属性

```python
@property
def num_completion_tokens(self):
    return self.num_tokens - self.num_prompt_tokens

@property
def num_blocks(self):
    return (self.num_tokens + self.block_size - 1) // self.block_size  # 向上取整

@property
def last_block_num_tokens(self):
    return self.num_tokens - (self.num_blocks - 1) * self.block_size
```

`num_blocks` 是 PagedAttention 的关键——它决定了每个序列需要多少个 KV-Cache Block。`last_block_num_tokens` 用于 decode 阶段确定最后一个 token 在哪个 slot 写入。

### block(i) 方法

```python
def block(self, i):
    assert 0 <= i < self.num_blocks
    return self.token_ids[i*self.block_size: (i+1)*self.block_size]
```

返回第 i 个 Block 对应的 token_ids 切片。BlockManager 用它来计算哈希和分配存储。

## 序列化支持

```python
def __getstate__(self):
    last_state = self.last_token if not self.is_prefill else self.token_ids
    return (self.num_tokens, self.num_prompt_tokens,
            self.num_cached_tokens, self.num_scheduled_tokens,
            self.block_table, last_state)

def __setstate__(self, state):
    self.num_tokens, self.num_prompt_tokens, self.num_cached_tokens, \
        self.num_scheduled_tokens, self.block_table, last_state = state
    if isinstance(last_state, list):
        self.token_ids = last_state
        self.last_token = self.token_ids[-1]
    else:
        self.token_ids = []
        self.last_token = last_state
```

这个序列化实现是 Tensor Parallel 通信的关键。在 TP 模式下，Sequence 对象通过 `pickle` → SharedMemory 传递给 worker 进程：

- **Prefill 状态**：传输完整 `token_ids`（因为所有 token 池要被 Prefill）
- **Decode 状态**：只传输 `last_token`（只需要最新一个 token）

这是一个非常聪明的空间优化：Decode 阶段每步只需传 1 个 int 而不是整个 token_ids 列表。

## Sequence 在调度中的使用

```
1. add_request(prompt) → Sequence(prompt, sp) → waiting 队列
2. schedule() → 读取 num_cached_tokens、num_tokens 决定分配多少 token
3. postprocess() → append_token() 追加新生成的 token
4. FINISHED → completion_token_ids 返回给用户
```
