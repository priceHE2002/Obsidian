# 模型执行引擎总览

模型执行引擎负责将调度器选出的序列实际跑在 GPU 上。它由两个文件构成：

| 文件 | 职责 | 行数 |
|------|------|------|
| `llm_engine.py` | 主循环：`generate()` 收敛、`step()` 编排 | ~110 |
| `model_runner.py` | GPU 推理：输入准备、CUDA Graph、TP 通信 | ~340 |

## 执行流程

```
LLMEngine.step()
    │
    ├─ scheduler.schedule()              ← 调度器决定本轮序列
    │
    ├─ model_runner.call("run", seqs, is_prefill)  ← GPU 推理
    │   ├─ [rank 0] prepare_prefill()/decode()     ← 准备输入 tensor
    │   ├─ [rank 0] write_shm() → worker 进程      ← TP 广播
    │   ├─ run_model()                              ← 前向推理
    │   │   ├─ Prefill 或 enforce_eager: model.forward()
    │   │   └─ Decode + CUDA Graph: graph.replay()
    │   └─ [rank 0] sampler(logits, temperatures)   ← 采样
    │
    └─ scheduler.postprocess()           ← 更新序列状态
```

## Tensor Parallel 通信模型

nano-vllm 的 TP 实现非常独特——它不使用 `torchrun`，而是通过 `multiprocessing` + `SharedMemory` + `pickle` 实现进程间通信：

```
Rank 0 (主进程)：
  - 运行 LLMEngine + Scheduler
  - write_shm(method_name, *args)  → 将方法名和参数序列化到 SharedMemory
  - 通过 Event 通知 Worker 进程
  - 调用方法，收集结果

Rank 1~N (Worker 进程)：
  - loop() 等待 Event
  - read_shm() 反序列化方法名和参数
  - 调用同名方法（主要是 "run" 和 "exit"）
  - dist.all_reduce / dist.gather 参与 TP 计算
```

这种设计的优势是：无需 `torchrun` / `torch.distributed.launch`，单个 Python 进程即可启动多 GPU 推理。
