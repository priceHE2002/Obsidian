# 工具与辅助

`utils/` 目录包含两个辅助模块：`context.py`（全局上下文传递）和 `loader.py`（HuggingFace 权重加载）。

## context.py — 全局上下文

这是 nano-vllm 最独特的设计模式之一。它用模块级全局变量替代了传统的参数传递。

### Context 数据结构

```python
@dataclass(slots=True)
class Context:
    is_prefill: bool = False
    cu_seqlens_q: torch.Tensor | None = None     # Prefill: Q 的累积序列长度
    cu_seqlens_k: torch.Tensor | None = None      # Prefill: K 的累积序列长度
    max_seqlen_q: int = 0                          # Prefill: 最大 Q 序列长度
    max_seqlen_k: int = 0                          # Prefill: 最大 K 序列长度
    slot_mapping: torch.Tensor | None = None       # token → KV-Cache slot 映射
    context_lens: torch.Tensor | None = None       # Decode: 每个序列的有效长度
    block_tables: torch.Tensor | None = None       # PagedAttention block table
```

### 全局读写

```python
_CONTEXT = Context()

def get_context():
    return _CONTEXT

def set_context(is_prefill, cu_seqlens_q=None, ..., block_tables=None):
    global _CONTEXT
    _CONTEXT = Context(is_prefill, cu_seqlens_q, ..., block_tables)

def reset_context():
    global _CONTEXT
    _CONTEXT = Context()
```

### 为什么用全局变量？

传统方法是在模型各层的 `forward` 中加入这些参数：

```python
def forward(self, x, slot_mapping, context_lens, block_tables, is_prefill, ...):
```

但这有严重问题：
1. **所有中间层都要透传这些参数**，即使它们自己不用（如 MLP 层）
2. **CUDA Graph 捕获困难**：这些参数在每次 replay 时变化，如果作为函数参数，CUDA Graph 无法捕获
3. **与 HuggingFace 模型签名不兼容**：无法直接使用 HuggingFace 的 `AutoModel` 设计

全局 Context 方案让模型层的 forward 保持简洁（如 `Attention.forward(q, k, v)`），仅在需要时通过 `get_context()` 获取参数。CUDA Graph 捕获时，这些参数作为静态 tensor 的填充值，在 replay 前通过索引更新而不改变 tensor 大小。

## loader.py — 权重加载器

```python
def load_model(model, path):
    packed_modules_mapping = getattr(model, "packed_modules_mapping", {})
    for file in glob(os.path.join(path, "*.safetensors")):
        with safe_open(file, "pt", "cpu") as f:
            for weight_name in f.keys():
                # 检查是否需要合并映射
                for k in packed_modules_mapping:
                    if k in weight_name:
                        v, shard_id = packed_modules_mapping[k]
                        param_name = weight_name.replace(k, v)
                        param = model.get_parameter(param_name)
                        weight_loader = getattr(param, "weight_loader")
                        weight_loader(param, f.get_tensor(weight_name), shard_id)
                        break
                else:
                    param = model.get_parameter(weight_name)
                    weight_loader = getattr(param, "weight_loader", default_weight_loader)
                    weight_loader(param, f.get_tensor(weight_name))
```

### 加载流程

1. 扫描模型目录下所有 `.safetensors` 文件
2. 对每个权重名，先检查是否需要通过 `packed_modules_mapping` 合并（如将 HF 的 `q_proj, k_proj, v_proj` 合并到单一 `qkv_proj`）
3. 如果不需要合并，直接用权重名查找对应的 `nn.Parameter`
4. 调用参数的 `weight_loader` 方法进行加载（支持 TP 分片）

### 设计的优雅之处

`safe_open(..., "cpu")` 在 CPU 上加载权重，然后 `param.data.copy_(loaded_weight)` 直接拷贝到 GPU。这避免了先用 `torch.load` 加载整个文件到内存的峰值占用。

`weight_loader` 绑定到参数上的模式让 loader 保持简洁——它不需要知道参数的类型和分片策略，只需调用绑定的方法。每种线性层（ColumnParallel、RowParallel 等）通过自己的 `weight_loader` 处理分片逻辑。

### default_weight_loader

```python
def default_weight_loader(param, loaded_weight):
    param.data.copy_(loaded_weight)
```

对于不涉及并行切分的参数（如 RMSNorm 的 weight），直接用默认 loader 进行完整拷贝。注意这里用的是 `copy_` 而非 `param.data = loaded_weight`——后者会创建一个新的 Tensor 并断开与 `nn.Parameter` 的关联。
